## 1. 系统概览与快速开始

LLM Wiki 是一个面向制造运营场景的 **AI 知识库 + 时序预测** 一体化平台，核心由四部分组成：

| 能力 | 说明 | 入口 |
|------|------|------|
| 知识管理 | 文档/手册/规程的结构化存储，支持分类、标签、状态生命周期 | 左侧菜单「知识管理」 |
| 智能问答 | 基于 Skill 路由 + RAG 检索增强的流式问答 | 左侧菜单「智能问答」 |
| 时序预测 | 内置 Chronos / TimesFM / ARIMA / Prophet 等多模型预测 | 左侧菜单「趋势分析」 |
| 系统管理 | Skill 配置、模型配置、Token 统计、调用日志 | 左侧菜单「系统管理」 |

**首次使用流程：**

1. 启动后端（`python -m uvicorn app.main:app --port 8001`）与前端（`npm run dev`）。
2. 后端启动时会自动执行 `seed_service.seed_initial_data()`，初始化演示数据（详见第 2 节）。
3. 浏览器打开 `http://localhost:5173`，登录后即可看到预置的数据集、知识条目、Skill 配置。
4. 按本说明书第 3–10 节的顺序体验各功能。

## 4. 数据集管理

### 4.1 数据集列表与创建

**入口：** 左侧菜单「数据集管理」→「数据集列表」。

**操作：**

| 操作 | 步骤 |
|------|------|
| 查看数据集 | 列表点击数据集名称，进入详情查看时序数据点、图表预览 |
| 新建数据集 | 点击「新建」→ 填写名称、频率（daily/weekly/monthly/quarterly/other）、单位、描述 → 在表格中录入或从 CSV/Excel 粘贴 `(time, value, label)` 数据点 → 保存 |
| 编辑数据集 | 列表点击「编辑」，可修改元信息与数据点 |
| 删除数据集 | 列表点击「删除」（会一并删除其下所有协变量） |

**实例演示：** 列表中应能看到「某车间月度产量（示例）」与「某设备季度故障次数（示例）」两条记录，分别有 24 与 12 个数据点。

### 4.2 协变量管理（重点）

协变量（外生变量 exog）是 ARIMA/Prophet 等统计模型在预测时引入的额外特征，例如节假日、促销、趋势项等。系统支持三种类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| `continuous` 连续型 | 数值连续 | trend（0,1,2,…）、month_sin |
| `binary` 二元型 | 0 或 1 | is_holiday、promotion |
| `categorical` 分类型 | 离散类别 | 季节（春/夏/秋/冬） |

**入口：** 数据集列表 → 点击某数据集的「协变量」按钮，进入协变量管理页。

#### 4.2.1 自动生成协变量

点击页面顶部「**自动生成协变量**」按钮，系统依据数据集频率自动生成以下协变量：

| 频率 | 自动生成的协变量 |
|------|-----------------|
| monthly | `trend`、`is_holiday`（中国节假日）、`month_sin`、`month_cos` |
| quarterly | `trend`、`quarter_sin`、`quarter_cos` |
| weekly / daily | `trend`、`is_holiday`、`dow_sin`、`dow_cos`（周周期编码） |

- 已存在的协变量（按 code 去重）会被跳过，可重复执行。
- 生成逻辑位于 [covariate_service.py](../backend/app/services/covariate_service.py) `auto_generate_covariates()`。

**实例演示：** 进入「某车间月度产量（示例）」的协变量页，应已存在 4 个自动生成的协变量（trend / is_holiday / month_sin / month_cos）+ 1 个手动协变量 promotion。点击「自动生成协变量」会提示「已跳过已存在的协变量」。

#### 4.2.2 新增协变量（手动）

点击「**新增协变量**」按钮，弹出编辑弹窗：

| 字段 | 说明 |
|------|------|
| 名称 | 中文显示名，如「促销活动」 |
| code | 英文标识，如 `promotion`，用于模型 exog 矩阵列名 |
| 类型 | 选择 continuous / binary / categorical |
| 描述 | 选填，说明业务含义 |
| 值列表 | 表格形式录入每行 `{time, value}`，time 必须与数据集时间点对齐 |

**值列表快捷操作：**
- 「**生成未来时间点**」：根据数据集频率自动追加未来 N 个时间点（值留空），用于预测时提供未来协变量值。例如月度数据集可生成 2025-01、2025-02…
- 「**清除未来值**」：一键移除所有超出数据集末尾时间点的行。
- 单行「删除」：移除指定行。

**实例演示（已内置）：** 月度产量数据集有一条手动协变量「促销活动」(`promotion`)，类型 binary，在 2023-07、2023-08、2023-12、2024-07、2024-08、2024-12 这 6 个点值为 1.0，其余为 0.0。可打开「查看值」抽屉核对。

#### 4.2.3 编辑 / 查看值 / 删除

| 操作 | 说明 |
|------|------|
| 编辑 | 修改名称、code、类型、描述、值列表 |
| 查看值 | 抽屉中只读查看完整值列表与统计 |
| 删除 | 二次确认后删除（不可恢复） |

#### 4.2.4 预览对齐矩阵

点击顶部「**预览对齐矩阵**」按钮，弹出对齐矩阵抽屉，展示一张表格：

- 每行 = 数据集的一个时间点（含未来时间点）
- 每列 = 一个协变量
- 末列 = 原始 value（目标值）

对齐策略（[covariate_service.py](../backend/app/services/covariate_service.py) `align_covariate()`）：
1. **精确匹配** time 字符串完全相等；
2. **日期前缀匹配** 如数据集点 `2024-01` 与协变量点 `2024-01-15`，按前缀匹配；
3. **缺失填 0**（binary/continuous）或保留空（categorical）。

**使用建议：** 在趋势分析启用「使用协变量」前，先在此预览确认所有协变量都已正确对齐，尤其是手动新增的协变量未来值是否齐全。

## 5. 趋势分析（每个按钮的使用）

**入口：** 左侧菜单「趋势分析」。

页面分四个区域：① 顶部工具栏 ② 预测结果图表 ③ STL 分解卡片 ④ 历史任务列表。此外有 5 个抽屉承载高级分析。

### 5.1 顶部工具栏配置

| 控件 | 说明 | 示例值 |
|------|------|--------|
| 数据集 | 选择要预测的数据集 | 某车间月度产量（示例） |
| 预测模型 | 选择 AI 模型（Chronos / TimesFM）或「统计模型」 | Chronos-2 (本地) |
| 预测模式 | 「未来预测」预测未知未来 / 「回测」用历史尾部验证 | 未来预测 |
| 训练点数 | 用于训练的历史数据点数（0 表示全部） | 0 |
| 预测步数 | 预测未来多少个点 | 6 |
| AI分析（checkbox） | 预测后是否调用 LLM 生成解读报告 | 勾选 |
| 使用协变量（checkbox） | ARIMA/Prophet 是否带入 exog | 勾选 |

### 5.2 主要预测按钮

#### 「开始预测」

最常用的入口按钮。点击后按当前工具栏配置执行一次预测：
- 若模型为 AI（Chronos/TimesFM），调用对应推理服务 `/predict`。
- 若模型为「统计模型」，等同于使用下方「统计预测」的默认算法（ARIMA）。
- 预测完成后图表区展示历史曲线 + 预测曲线 + 置信区间。
- 勾选「AI分析」会额外生成一段 LLM 解读。
- 勾选「使用协变量」时，ARIMA/Prophet 会自动调用 `build_exog_matrix()` 拼接 exog_train + exog_future。

**实例演示：** 选「某车间月度产量（示例）」+ Chronos-2 + 未来预测 + 预测步数 6 + 勾选 AI分析 → 点「开始预测」→ 图表显示 2025-01 ~ 2025-06 的产量预测。

#### 「统计预测」

下拉/分组按钮，包含 4 个子项：**ARIMA / ETS / Theta / Prophet**。点击任一算法单独执行一次统计预测。

- 不依赖 GPU 推理服务，纯 CPU 即可运行，适合无 Chronos/TimesFM 环境时演示。
- 启用「使用协变量」时：ARIMA → ARIMAX、Prophet → Prophet with add_regressor。

**实例演示：** 选「某车间月度产量（示例）」+ 勾选「使用协变量」+ 点「统计预测 → Prophet」→ 预测结果下方「协变量列表」会显示 `trend, is_holiday, month_sin, month_cos, promotion`，证明 exog 已生效。

#### 「使用协变量」checkbox

- 仅对统计模型（ARIMA/Prophet）生效。
- 勾选前请先在数据集管理页配置协变量，并在「预览对齐矩阵」确认对齐无误。
- 若协变量未来值缺失，Prophet 会按 0 填充，ARIMAX 会报错提示。
- 鼠标悬停 tooltip：「启用后 ARIMA/Prophet 会使用数据集的协变量（exog）增强预测。请在数据集管理页配置协变量。」

#### 「AI分析报告」

点击后弹出抽屉，调用 LLM 对最近一次预测结果生成解读报告，内容包含：
- 趋势判断（上升 / 下降 / 平稳）
- 季节性与周期识别
- 异常点提示
- 业务建议

> 该按钮依赖已激活的 LLM 模型配置。

### 5.3 高级分析按钮（5 个抽屉）

#### 「交叉验证」

- 点击「交叉验证」打开抽屉 → 设置切分方式（如 expanding window / rolling window）与切分次数 → 点「执行交叉验证」。
- 结果展示：每次切分的 MAE / RMSE / MAPE 明细表 + 平均指标行 + MAE 趋势图。
- 用于评估预测模型在不同历史窗口上的稳定性。

#### 「多模型对比」

- 点击「多模型对比」打开抽屉 → 勾选要对比的模型（如 Chronos / ARIMA / Prophet / Theta）→ 点「执行多模型对比」。
- 结果展示：各模型在同一预测窗口下的指标对比表（MAE/RMSE/MAPE）+ 预测曲线叠加图。
- 用于选型决策。

#### 「自动选模型」

- 点击「自动选模型」打开抽屉 → 点执行 → 系统对多个候选模型跑交叉验证，按综合指标排序输出最优模型。
- 结果展示：模型排名表 + 推荐模型高亮。
- 空状态提示：「点击「自动选模型」按钮开始评估」。

#### 「集成预测」

- 点击「集成预测」按钮 → 系统对多个模型分别预测后做加权融合（如平均 / 按误差倒数加权）→ 输出集成预测曲线。
- 完成后提示「集成预测完成：N 个模型参与」。
- 适合单模型都不够稳定时降低方差。

#### 「特征工程」

- 点击「特征工程」打开「高级特征工程」抽屉 → 展示从序列中提取的特征（滞后项、滑动统计、傅里叶项、趋势/季节分解项等）。
- 用于理解模型可用特征、辅助调参。

#### 「异常检测」

- 点击「异常检测」打开「统计异常检测」抽屉 → 选择方法（STL残差 / 移动平均残差 / IQR）→ 执行。
- 结果在图表上高亮标注异常点，并输出异常点列表（时间、值、残差、是否异常）。
- 用于发现历史数据中的突发事件（如设备故障导致的产量骤降）。

#### 「超参优化」

- 点击「超参优化」按钮 → 对所选模型执行网格/贝叶斯超参搜索（如 ARIMA 的 p/d/q，Prophet 的 changepoint_prior_scale）。
- 结果输出最优参数组合与对应验证指标。
- 执行较慢，建议在小数据集上试运行。

### 5.4 STL 季节性分解

页面中部「季节性分解（STL）」卡片：

- 点「**生成分解**」按钮 → 对当前数据集执行 STL 分解。
- 输出三张子图：趋势项 (Trend) + 季节项 (Seasonal) + 残差项 (Residual)。
- 空状态提示：「点击「生成分解」按钮查看 STL 季节性分解结果」；数据量不足时显示「数据量不足，无法进行季节性分解」。

**实例演示：** 对「某车间月度产量（示例）」点「生成分解」，季节项会清晰显示 7-8 月旺季高峰、12 月年终高峰。

### 5.5 历史任务列表

页面底部展示历次预测任务，每条记录可：
- **查看**：回填图表为该次任务结果。
- **导出 Excel**：导出历史 + 预测数据为 .xlsx。
- **导出 CSV**：导出为 .csv 供二次分析。
- **删除**：清理历史任务。

## 6. 智能问答与 Skill 路由（重点）

**入口：** 左侧菜单「智能问答」。

### 6.1 问答交互

- 输入框输入问题 → 回车或点发送 → 答案以 **SSE 流式**逐字输出。
- 答案中引用的知识来源以角标 `[1][2]` 标注，点击角标可展开来源卡片（标题、片段、相似度）。
- 答案底部可：「反馈」（👍/👎）、点击「推荐追问」继续对话。
- 当未配置 LLM 或检索失败时，显示降级提示并给出原始检索片段。

### 6.2 Skill 三级匹配机制

每次提问后端执行 [skill_router.py](../backend/app/services/skill_router.py) `route(question, db)`，三级匹配按优先级依次执行：

```
1. 关键词 / 正则匹配（match_type = "keyword"）
   ├─ 遍历所有 enabled=True 的 Skill
   ├─ 对每个 Skill 的 trigger_keywords 做子串包含判断
   ├─ 对 trigger_patterns 做正则 search 判断
   ├─ 命中次数 > 0 的 Skill 参与排序
   └─ 取命中次数最多者；同分取第一个
           ↓ 未命中
2. 语义匹配（match_type = "semantic"）
   ├─ 将问题 Embedding（bge-m3）
   ├─ 每个 Skill 的 description Embedding（md5 缓存，避免重复向量化）
   ├─ 计算余弦相似度
   └─ 最高分 > 0.7 才命中，否则放弃
           ↓ 未命中
3. 默认兜底（match_type = "default"）
   └─ 取 is_default=True 的 Skill；不存在则取第一个 Skill
```

命中 Skill 后：
- 加载该 Skill 的 `prompt_template`（含 `{context}` `{question}` 占位符）。
- 按 Skill 的 `knowledge_scope` 限定 RAG 检索范围（见 6.3）。
- 调用 LLM 流式生成答案。

### 6.3 knowledge_scope 知识范围限定

Skill 的 `knowledge_scope` 字段（JSON）形如：

```json
{"category_ids": [4], "tag_ids": [1]}
```

含义：检索知识时，**只在该 Skill 指定分类 ∪ 标签范围内**检索。
- `category_ids`：限定知识条目的 `category_id` ∈ 此列表。
- `tag_ids`：限定知识条目带有此列表中至少一个标签。
- 两者为「或」关系（取并集），任一为空则该维度不限。
- `{}`（空对象）= 不限范围，检索全库（如「通用问答」Skill）。

**实例对照：** 故障诊断 Skill 的 `knowledge_scope = {"category_ids":[4], "tag_ids":[1]}`，意味着只会检索「设备管理」分类下或带「故障代码」标签的知识。因此「CNC加工中心E01报警故障诊断与处理指南」（分类=设备管理、标签=故障代码）会被检索到，而「数控加工工艺参数规范」（分类=制造运营）不会被检索到。

### 6.4 实例问题对照表

下列问题可直接在「智能问答」页输入测试，验证 Skill 路由与知识匹配：

| 输入问题 | 命中 Skill | 匹配类型 | 命中依据 | 检索到的知识 |
|---------|-----------|---------|---------|-------------|
| CNC加工中心E01报警怎么处理？ | 故障诊断 | keyword | 关键词「E01」「故障」（隐含）+ 正则 `E\d+` | CNC加工中心E01报警故障诊断与处理指南 |
| 设备定期保养维护规程是什么？ | 保养维护 | keyword | 关键词「保养」「维护」「规程」 | 设备定期保养维护规程与周期表 |
| 数控加工工艺参数怎么选？ | 工艺指导 | keyword | 关键词「工艺」「参数」「加工」 | 数控加工工艺参数规范与操作要点 |
| 产品质量检验合格判定规则？ | 质量检验 | keyword | 关键词「质量」「检验」「合格」「标准」 | 产品质量检验标准与不合格品判定规则 |
| MOM系统怎么录入生产工单？ | 通用问答 | default | 无关键词命中、语义分 ≤ 0.7 → 兜底 | MOM系统操作常见问题FAQ |

> **如何验证路由结果？** 在「Skill 管理」→ 选某 Skill → 点「测试路由」→ 输入问题，返回 `match_type`、`score`、`all_scores`（所有 Skill 的关键词命中数与语义得分）。详见第 8.5 节。

## 7. 知识管理

**入口：** 左侧菜单「知识管理」。

### 7.1 列表与筛选

- 支持按分类、标签、状态（draft/published/archived）、关键词筛选。
- 关键词对 title 和 content 做 LIKE 模糊匹配（特殊字符已转义）。
- 分页默认 10 条/页。

### 7.2 创建 / 编辑

| 字段 | 说明 |
|------|------|
| 标题 | 必填，≤200 字 |
| 内容 | Markdown 格式，支持表格、代码块、列表 |
| 内容格式 | markdown / text / html |
| 分类 | 可选，关联「分类管理」中的目录 |
| 标签 | 多选，关联「标签管理」 |
| 状态 | draft（草稿，不索引）/ published（发布，参与检索）/ archived（归档，不索引） |

### 7.3 状态生命周期

```
   ┌─────────┐  发布   ┌───────────┐  归档   ┌──────────┐
   │  draft  │ ──────► │ published │ ──────► │ archived │
   └─────────┘         └───────────┘         └──────────┘
        ▲                    │ 重新发布           │ 恢复
        └────────────────────┴───────────────────┘
```

- **draft → published**：触发向量索引重建，开始可被检索。
- **published → draft/archived**：移除向量索引，立即不可检索。
- **archived → published**：重建索引。
- 每次状态变更写入 `KnowledgeStatusLog` 审计日志（含操作人、原因、时间），可在知识详情页查看。

> 答案缓存：知识增删改后自动清空 `answer_cache`，防止返回过期答案。

### 7.4 文件上传

- 支持批量上传 .txt / .md / .pdf / .docx / .xlsx / .csv 等。
- 上传后自动解析为 Markdown，创建知识条目并建立向量索引。
- **去重**：按文件 SHA-256 哈希去重，重复文件跳过并提示。
- **自动打标签**：勾选后 LLM 分析内容前 2000 字，从现有标签优先匹配，不足时新建 1-3 个标签。

### 7.5 分类与标签

- **分类管理**：树形结构，支持多级（parent_id），用于知识归类与 Skill knowledge_scope 限定。
- **标签管理**：扁平结构，带颜色，用于跨分类的知识关联与检索过滤。

### 7.6 状态变更日志

知识详情页可查看该条知识的全部状态变更记录（操作人、旧/新状态、原因、时间），用于合规审计。

## 8. Skill 管理

**入口：** 左侧菜单「系统管理」→「Skill 管理」。

### 8.1 列表与创建

- 列表展示所有 Skill，可按 category / function / enabled 筛选。
- 可从模板创建（`/api/skills/from-template`），或新建空白 Skill。

### 8.2 关键字段

| 字段 | 说明 | 示例 |
|------|------|------|
| name | Skill 名称 | 故障诊断 |
| description | 描述（用于语义匹配 Embedding） | 针对设备故障、报警、错误代码进行诊断… |
| category | 模块维度 | 设备管理 / 制造运营 / 质量管理 / 通用 |
| function | 功能维度 | 故障诊断 / 保养维护 / 工艺指导 / 质量检验 / 通用问答 |
| trigger_keywords | 关键词 JSON | `["故障","报错","E0","E1"]` |
| trigger_patterns | 正则 JSON | `["E\\d+","故障代码.*"]` |
| prompt_template | Prompt 模板（含 `{context}` `{question}`） | 见下 |
| knowledge_scope | 知识范围 JSON | `{"category_ids":[4],"tag_ids":[1]}` |
| enable_query_rewrite | 是否启用查询改写（多轮指代消解） | false |
| context_turns | 查询改写时取最近 N 轮对话 | 3 |
| enabled | 启用状态 | true |
| is_default | 是否默认兜底 Skill | false（仅「通用问答」为 true） |

### 8.3 Prompt 模板变量

模板中可使用两个占位符：
- `{context}`：RAG 检索到的知识片段（按 Skill 的 knowledge_scope 限定范围检索后拼接）。
- `{question}`：用户当前问题（若启用查询改写，则为改写后的问题）。

**实例（故障诊断 Skill 模板）：**

```
你是MOM系统设备故障诊断专家。请根据设备故障现象和知识库内容，给出诊断结论和处理建议。

知识上下文：
{context}

用户问题：{question}

请按以下结构回答：
1. 故障现象分析
2. 可能原因
3. 处理建议
4. 注意事项
```

### 8.4 knowledge_scope 配置

编辑 Skill 时在 knowledge_scope 字段填入 JSON：

- `{"category_ids":[4], "tag_ids":[1]}`：检索「设备管理」分类 ∪ 带「故障代码」标签的知识。
- `{"category_ids":[1]}`：只检索「制造运营」分类。
- `{}`：不限范围，检索全库（默认 Skill 用）。

### 8.5 测试路由

**入口：** Skill 列表 → 选某 Skill → 详情页点「测试路由」（实际调用 `POST /api/skills/{sid}/test`）。

> 注意：端点路径中的 `sid` 是必传路径参数（指向任一已存在 Skill 即可，测试本身会跑全量路由），不是 `POST /api/skills/test-route`。

输入测试问题后返回：

```json
{
  "matched_skill": { "id": 1, "name": "故障诊断", ... },
  "match_type": "keyword",        // keyword / semantic / default
  "score": 2.0,                  // keyword=命中次数; semantic=相似度; default=0
  "all_scores": [
    { "skill_id": 1, "skill_name": "故障诊断", "keyword_hits": 2, "semantic_score": 0.8231 },
    { "skill_id": 2, "skill_name": "保养维护", "keyword_hits": 0, "semantic_score": 0.5123 },
    ...
  ]
}
```

可用于：调试关键词是否命中、对比语义得分、验证兜底逻辑。

## 9. 模型配置

**入口：** 左侧菜单「系统管理」→「模型配置」。

支持四类模型，每类同一时间仅一个 `is_active=True`：

| 类型 | 用途 | 推荐模型 |
|------|------|---------|
| LLM | 问答生成、AI分析报告、自动打标签 | DeepSeek-V4-Flash |
| Embedding | 向量化（知识索引、Skill 语义匹配、RAG 检索） | bge-m3 |
| Forecast | 时序预测推理 | Chronos-2 / TimesFM 2.5（本地 8501/8502） |
| Rerank | RAG 检索结果重排序 | bge-reranker |

**操作：**
- 新增：填写名称、类型、API URL、API Key（加密存储）、model_name。
- 编辑 / 删除 / 切换激活状态。
- Forecast 本地推理服务部署见 [chronos-deploy.md](./chronos-deploy.md)、[timesfm-deploy.md](./timesfm-deploy.md)。

- 数据备份与导入导出。

## 附录：常见问题

**Q1：趋势分析点「开始预测」无响应？**
检查「模型配置」中是否已激活 Forecast 模型（Chronos/TimesFM），且本地推理服务（8501/8502）已启动；或改用「统计预测」按钮（纯 CPU 可运行）。

**Q2：智能问答提示「没有可用的 Skill」？**
进入「Skill 管理」确认至少有一个 `enabled=True` 的 Skill；系统默认会预置 5 个 Skill，若被误删可重装或手动新建。

**Q3：勾选「使用协变量」后 ARIMA 报错？**
通常是协变量未来值缺失。进入「数据集管理 → 协变量 → 预览对齐矩阵」，确认未来时间点的协变量值都已填写；缺失的可在编辑弹窗用「生成未来时间点」补齐后手动填值。

**Q4：知识上传后问答检索不到？**
检查知识状态是否为 `published`（draft/archived 不会被索引）；检查 Skill 的 `knowledge_scope` 是否把该知识的分类/标签包含在内。

**Q5：Skill 路由总命中「通用问答」？**
说明关键词和语义都没命中。去「Skill 管理 → 测试路由」输入问题，查看 `all_scores` 中各 Skill 的 `keyword_hits` 与 `semantic_score`，据此补充 trigger_keywords 或调整 description（语义匹配基于 description）。
